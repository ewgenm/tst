## 1. АРХИТЕКТУРА И ПРИНЦИПЫ

### 1.1. Слои приложения
```
app/
├── Http/
│   ├── Controllers/Api/     # Тонкие контроллеры (вызов сервисов)
│   ├── Requests/            # FormRequest валидация
│   ├── Resources/           # API Resources (JSON трансформация)
│   └── Middleware/          # EnsureJsonResponse, Sanctum, CORS
├── Models/                  # Eloquent модели (с отношениями, скоупами)
├── Policies/                # Авторизация (матрица ACL из ТЗ №0)
├── Services/                # Бизнес-логика (чистые классы)
├── Actions/                 # Single-responsibility действия
├── Events/                  # WebSocket события (ShouldBroadcastNow)
├── Jobs/                    # Очереди (уведомления, AI)
├── Rules/                   # Кастомные правила валидации
└── Traits/                  # Переиспользуемая логика
```

### 1.2. Принципы кода
*   **API-First:** Все ответы через `API Resource`, формат `{ success, data, meta }`.
*   **DELETE returns 204:** Успешное удаление возвращает `response()->noContent()` (204).
*   **Type-safe:** PHP 8.3 typed properties, `readonly class` для DTO, enum для статусов.
*   **Thin Controllers:** Контроллеры только принимают запрос, валидируют, вызывают сервис, возвращают ресурс.
*   **Service Layer:** Бизнес-логика вынесена в `Services/` и `Actions/`.
*   **Policy-Based ACL:** Все проверки прав через `Policies`, с явной проверкой `owner_id`.
*   **Test-Driven:** Каждый эндпоинт покрывается `Feature Test` перед генерацией следующего.

### 4.2. Ключевая логика
*   **Создание проекта:** `owner_id = auth()->id()`, автоматическое добавление создателя как `ProjectMember` с ролью `admin` и статусом `active`.
*   **ACL — явная проверка owner_id:** `ProjectPolicy` должен явно проверять `owner_id` отдельно от ролей в `project_members`.
*   **Приглашение:** `InviteMemberAction` создает `ProjectMember` со `status='pending'`, создает уведомление в таблице `notifications` с типом `project_invite`, отправляет email (queue) и real-time событие через Reverb.
*   **Принятие приглашения:** `AcceptInviteAction` меняет `status` на `active`. Доступно только приглашенному пользователю.
*   **Сортировка:** Поле `sort_order` обновляется через `PUT /projects/{id}`.
*   **Задачи проекта:** `GET /projects/{id}/tasks` — это алиас на `GET /tasks?project_id={id}`.

### 4.3. ProjectPolicy (пример кода)
```php
class ProjectPolicy
{
    public function view(User $user, Project $project): bool
    {
        // Владелец ИЛИ участник с active статусом
        return $project->owner_id === $user->id 
            || $project->members()->where('user_id', $user->id)->where('status', 'active')->exists();
    }
    
    public function delete(User $user, Project $project): bool
    {
        // ТОЛЬКО владелец может удалить проект
        return $project->owner_id === $user->id;
    }
    
    public function archive(User $user, Project $project): bool
    {
        // Владелец ИЛИ Admin
        return $project->owner_id === $user->id 
            || $this->isAdmin($user, $project);
    }
    
    public function invite(User $user, Project $project): bool
    {
        // Владелец ИЛИ Admin
        return $project->owner_id === $user->id 
            || $this->isAdmin($user, $project);
    }
    
    private function isAdmin(User $user, Project $project): bool
    {
        return $project->members()
            ->where('user_id', $user->id)
            ->where('role', 'admin')
            ->where('status', 'active')
            ->exists();
    }
}
```

### 5.2. Ключевая логика
*   **Inbox:** Задачи с `project_id = null` автоматически попадают в Inbox (фильтр на фронтенде).
*   **Статусы:** Enum `TaskStatus` без `inbox` (только `todo`, `in_progress`, `review`, `done`).
*   **Каскадное удаление подзадач:** При удалении родительской задачи все подзадачи удаляются автоматически (`cascadeOnDelete()` на `parent_task_id`).
*   **Recurring:** `CompleteTaskAction`:
  ```php
  if ($task->is_recurring && $task->recurring_rule) {
      $nextDue = $this->calculateNextDue($task->due_at, $task->recurring_rule);
      $task->replicate()->fill([
          'status' => 'todo',
          'due_at' => $nextDue,
          'completed_at' => null,
      ])->save();
  }
  ```
*   **Sorting:** 
  - Поле `position` (integer). 
  - `PUT /tasks/{id}/position` с телом `{ "position": 5 }` или `{ "after_id": 123 }`.
  - При изменении `project_id` или `status` задача автоматически получает последнюю позицию в новом контексте.
*   **Aggregates:** В `TaskResource` использовать `whenLoaded()` для отношений. Counts через `withCount()` в контроллере.
*   **subtasks_total / subtasks_completed:**
  ```php
  // В TaskController index/show
  ->withCount(['subtasks', 'subtasks as subtasks_completed_count' => fn($q) => $q->where('status', 'done')])
  
  // В TaskResource
  'subtasks_total' => $this->subtasks_count,
  'subtasks_completed' => $this->subtasks_completed_count,
  ```

### 5.3. Валидация recurring_rule
**`app/Rules/ValidRecurringRule.php`:**
```php
class ValidRecurringRule implements ValidationRule
{
    public function validate(string $attribute, mixed $value, Closure $fail): void
    {
        if ($value && !preg_match('/^FREQ=(DAILY|WEEKLY|MONTHLY|YEARLY)(;INTERVAL=\d+)?(;BYDAY=[A-Z,]+)?$/', $value)) {
            $fail('Неверный формат правила повторения. Допустимы: FREQ=DAILY, FREQ=WEEKLY, FREQ=MONTHLY');
        }
    }
}
```

**В StoreTaskRequest / UpdateTaskRequest:**
```php
'recurring_rule' => 'nullable|string|new ValidRecurringRule()',
```

**LogCompletionAction логика:**
```php
public function execute(Habit $habit, Carbon $date = null): bool
{
    $date = $date ?? now();
    
    // Запрет на отметку за будущие даты
    if ($date->isFuture()) {
        throw new \Exception('Нельзя отметить привычку за будущие даты');
    }
    
    // Проверка на уже выполненное сегодня
    $exists = $habit->completions()
        ->whereDate('completed_date', $date)
        ->exists();
    
    if ($exists) {
        return false; // Уже отмечено
    }
    
    // Создание записи
    $habit->completions()->create(['completed_date' => $date]);
    
    // Обновление streak
    $this->updateStreak($habit);
    
    return true;
}
```

Broadcast::channel('project.{projectId}', function ($user, $projectId) {
    $project = \App\Models\Project::find($projectId);
    if (!$project) return false;
    
    // Владелец ИЛИ участник с active статусом
    return $project->owner_id === $user->id 
        || $project->members()
            ->where('user_id', $user->id)
            ->where('status', 'active')
            ->exists();
});

Broadcast::channel('user.{userId}', function ($user, $userId) {
    return $user->id === (int) $userId;
});

}

public function test_user_can_create_task_in_project(): void
{
    $user = User::factory()->create(['email_verified_at' => now()]);
    $project = Project::factory()->for($user, 'owner')->create();
    
    $response = $this->actingAs($user)
        ->postJson('/api/v1/tasks', [
            'title' => 'New Task',
            'project_id' => $project->id,
            'status' => 'todo',
            'priority' => 'medium',
        ]);
    
    $response->assertStatus(201)
        ->assertJsonStructure(['success', 'data' => ['id', 'title', 'project_id']]);
    
    $this->assertDatabaseHas('tasks', ['title' => 'New Task', 'project_id' => $project->id]);
}

public function test_subtasks_are_cascade_deleted(): void
{
    $user = User::factory()->create(['email_verified_at' => now()]);
    $parent = Task::factory()->for($user, 'creator')->create();
    $subtask = Task::factory()->for($user, 'creator')->create(['parent_task_id' => $parent->id]);
    
    $parent->delete();
    
    $this->assertDatabaseMissing('tasks', ['id' => $parent->id]);
    $this->assertDatabaseMissing('tasks', ['id' => $subtask->id]); // Каскадное удаление
}

public function test_delete_returns_204(): void
{
    $user = User::factory()->create(['email_verified_at' => now()]);
    $task = Task::factory()->for($user, 'creator')->create();
    
    $response = $this->actingAs($user)
        ->deleteJson("/api/v1/tasks/{$task->id}");
    
    $response->assertStatus(204);
}
